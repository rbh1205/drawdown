# Implementation Plan: Specialty Coffee Grind Tracker

**Branch**: `001-coffee-grind-tracker` | **Date**: 2026-05-06 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/001-coffee-grind-tracker/spec.md`

## Summary

A personal, offline-first Progressive Web App for tracking specialty coffee bags, grind settings, and collection stats. Built on SvelteKit (static adapter) with IndexedDB storage via `idb-keyval`. All data is local; no accounts, no server, no sync. The core value is the grind-setting drift tracker: an immutable per-bag history of dial-in values per grinder/brew-method combination, accessible in two taps from the home screen.

## Technical Context

**Language/Version**: TypeScript (opted in — 6 entity types with inter-references justify type safety)  
**Primary Dependencies**: SvelteKit (static adapter), idb-keyval, vite-plugin-pwa  
**Storage**: IndexedDB via `idb-keyval` — six named stores (bags, roasters, grinders, brew-methods, grind-settings, preferences)  
**Testing**: Vitest + @testing-library/svelte  
**Target Platform**: Android Chrome 110+ via PWA; desktop Chrome as secondary  
**Project Type**: Progressive Web App (installable, fully offline)  
**Performance Goals**: Home screen loads active bags instantly; stats screen renders in <2 s with 100+ bags; calculator computes in <1 s  
**Constraints**: Fully offline; all touch targets ≥44×44 dp; WCAG AA color contrast; no network requests at runtime  
**Scale/Scope**: Single user; dozens to hundreds of bags; ~5–20 grinders and brew methods; hundreds of grind setting entries

## Implementation Iterations

Each iteration is a vertical, independently testable slice. The agent stops after each iteration and provides a user-perspective summary. The next iteration does not begin until the user has verified the current one.

---

### Iteration 1 — PWA Shell + Active Bag List

**Delivers**: The user can install the app, see an empty home screen, add a coffee bag (name, origin, roaster, processing method), and have it appear in their rotation. The app works offline.

**Scope**:
- SvelteKit project with static adapter, TypeScript, vite-plugin-pwa configured
- Web app manifest + service worker (Workbox, cache-first for static assets)
- idb-keyval stores initialised (all six stores, preferences seeded)
- `Roaster` entity: create inline during bag add, list existing
- `Bag` entity: required fields only (name, roasterId, origin, processingMethod, status='active', dateAdded)
- Home screen: scrollable list of active bags (name + roaster), empty-state with add prompt
- Add-bag screen: form for required fields + roaster picker/create
- Bottom tab bar: four tabs (Home, Archive, Calculator, Stats) — Archive/Calculator/Stats show placeholder screens
- Mobile viewport: 44×44 dp tap targets, no layout shifts

**Test criteria** (user-driven):
- Install as PWA on Android or via DevTools.
- Kill network; reopen — app loads fully.
- Add a bag → it appears on the home screen.
- Add a second bag → both appear.

---

### Iteration 2 — Grind Settings

**Delivers**: The user can manage their grinders and brew methods, and log grind settings for any active bag. The full drift history is visible per combo.

**Scope**:
- `Grinder` entity: create, list, delete (blocked with explanation if settings exist)
- `BrewMethod` entity: create, list, delete (blocked if settings exist); 6 seed methods pre-populated
- `GrindSetting` entity: log new entry (bag + grinder + brew method + value + optional notes); immutable once saved
- Bag detail screen: grind settings grouped by grinder/brew method, each group showing history in reverse-chrono order
- "Log setting" flow: grinder selector → brew method selector → value input → save; previous history shown for reference before saving
- Grinder/brew-method management accessible from a settings icon on the relevant screens
- Empty states: no grinders yet, no brew methods yet, no settings yet for this combo

**Test criteria** (user-driven):
- Add a grinder and two brew methods.
- On an active bag, log a grind setting for each combo.
- Log a second setting for one combo — confirm both appear with dates, newest first.
- Try to delete a grinder with settings — confirm deletion is blocked with a message.

---

### Iteration 3 — Finish a Bag, Star Rating & Archive

**Delivers**: The user can mark a bag as finished (prompted for a star rating), and browse all archived bags with their grind history and rating. Archive supports filtering.

**Scope**:
- "Mark as Finished" action on active bag detail
- Rating prompt (1–5 star selector, skippable) shown before archiving
- Bag moves to archive: `status='archived'`, `dateFinished` set, `ratingScore` set (or null if skipped)
- Archive screen: scrollable list of finished bags, reverse-chrono by `dateFinished`; each entry shows name, roaster, rating stars, dates
- Archive filter bar: filter by origin, processing method, roaster (single active filter at a time)
- Archive bag detail: full grind settings history + rating (editable from this view)
- Star rating edit: tap stars on archive entry to update score (replaces previous)
- Home screen empty state when all bags archived

**Test criteria** (user-driven):
- Finish a bag with a 4-star rating → it disappears from home, appears in archive with stars.
- Finish a second bag, skip the rating → archives with no stars; add rating later from archive.
- Filter archive by processing method → confirm only matching bags shown.
- Tap archived bag → see full grind settings history.

---

### Iteration 4 — Full Bag Data Entry & Barcode

**Delivers**: All optional bag fields are supported (altitude, variety, roast date, barcode). Barcode scan via camera is available with manual fallback.

**Scope**:
- Bag form extended with optional fields: variety, altitude, roast date (date picker), barcode
- Barcode scan button: opens camera via `BarcodeDetector` API, detected code pre-fills barcode field; if `BarcodeDetector` unavailable, button replaced with manual text field
- Edit bag screen: all fields editable post-creation (FR-013)
- Roaster detail/edit: manage roaster name, country, website
- Roast date display on bag detail and archive entries (formatted, not raw ISO string)

**Test criteria** (user-driven):
- Add a bag with all fields including a scanned barcode — confirm barcode value saved.
- Edit an existing bag's origin — confirm change persists after app restart.
- On a device/browser without BarcodeDetector, confirm manual barcode entry works.

---

### Iteration 5 — Brew Ratio Calculator

**Delivers**: The standalone calculator tab works. The user sets their preferred ratio and it persists across restarts.

**Scope**:
- Calculator screen: dose input → water output (and reverse mode)
- Default ratio persisted in `preferences` store
- Ratio setter: inline on calculator screen (type a ratio like "16" for 1:16)
- Input validation: non-numeric inputs rejected with clear message
- Bidirectional: toggle between "I know the dose" and "I know the water" modes

**Test criteria** (user-driven):
- Enter 20 g dose with 1:16 ratio → confirm 320 g water output.
- Switch to reverse: enter 320 g water → confirm 20 g dose.
- Change ratio to 1:15 → close and reopen app → confirm ratio still 1:15.
- Enter letters in dose field → confirm error message, no crash.

---

### Iteration 6 — Stats Screen

**Delivers**: The stats screen shows aggregated insights across the full collection (active + archived).

**Scope**:
- Stats screen: processing method breakdown (counts per method), origins list (alphabetical with counts), varieties tried (count or list), altitude distribution (min/max/most common), roasters explored (count + list), bags per month (calendar-style streak view)
- All stats aggregate across active + archived bags
- Empty state for each stat section when no bags exist
- Stats are computed on-mount (no caching needed at this scale)

**Test criteria** (user-driven):
- With 5+ bags across different origins and processing methods: confirm each stat reflects the data correctly.
- Open stats with no bags logged — confirm appropriate empty states.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `idb-keyval` dependency | IndexedDB API without a wrapper requires ~50+ lines of boilerplate per store operation (open DB, transaction, store, cursor). idb-keyval reduces this to 3-line calls. | Raw IndexedDB is verbose and error-prone; `localStorage` cannot hold structured collections per constitution. |
| `vite-plugin-pwa` dependency | Service worker registration, Workbox precache config, and manifest wiring is ~200 lines of boilerplate. Plugin generates this from a 20-line config object. | Manual service worker authoring adds high maintenance burden for a solved problem. |
